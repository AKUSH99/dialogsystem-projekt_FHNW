## Project

Buy-Bot is a laptop purchasing advisor chatbot built for FHNW Dialogsysteme. It guides users through a structured intake, routes them to a domain-specific expert agent, queries a SQLite product database, and generates a natural-language recommendation. Full design rationale is in `vaagi_thought.md`.

## Architecture

The pipeline is a single LangGraph graph. All agents share one `BuyBotState` TypedDict (defined in `agents/state.py`). No inter-process communication, no temp files.

```
intake_agent → router → [uni|gaming|professional|office]_agent → search_agent → suggestion_agent
                                                                        ↑
                                                              qa_agent (always-on, any stage)
```

**Stage flow:**
1. `intake_agent` — collects 5 slots: `language`, `budget`, `preferred_os`, `use_case`, `mobility`
2. `router` — reads `use_case`, picks the correct expert agent
3. Expert agent — asks follow-up questions, writes enrichments to `state["user_profile"]`
4. `search_agent` — queries `data/laptops.db` using full state, writes `laptop_primary` + `laptop_alternative`
5. `suggestion_agent` — generates recommendation, adapts language to user's technical level
6. `qa_agent` — intercepts off-script questions at any stage, hands back to the interrupted stage

### Shared State

`BuyBotState` in `agents/state.py`:

```python
session_id: str          # LangGraph thread ID — groups all turns in LangSmith
messages: list           # full conversation history across all agents
current_stage: str       # "intake" | "expert" | "search" | "suggestion" | "done"
language: str | None     # "english" | "german" — chosen first, used by all agents
budget: float | None     # top-level — guaranteed after intake
preferred_os: str | None # "windows" | "macos" | "no_preference"
use_case: str | None     # "gaming" | "university" | "professional" | "office"
mobility: str | None     # "high" | "medium" | "low"
user_profile: dict       # expert agent enrichments — shape varies by use case
laptop_primary: dict | None
laptop_alternative: dict | None
```

The 5 intake slots are top-level typed fields (always same shape). Expert enrichments go into `user_profile` as a plain dict because the shape differs per use case — the search agent reads them with `.get()`.

### Prompt Structure

Every LLM call uses two layers (both defined in `agents/prompts.py`):
1. `BUYBOT_SYSTEM_PROMPT` — global persona, tone, language adaptation rules (same for every agent)
2. Agent-specific role prompt — what this agent's current job is

Changing Buy-Bot's personality means editing `prompts.py` once.

### LLM Access

All LLM calls go through OpenRouter (`https://openrouter.ai/api/v1`) using `ChatOpenAI` from `langchain-openai`.

Two models are used — configured in `main.py` and `frontend/app.py`:

| Model | Used by | Reason |
|---|---|---|
| `llm_fast` | intake, expert agents, classifier | slot extraction and follow-up questions — speed matters |
| `llm_strong` | QA agent, search ranking, suggestion | nuanced answers and final recommendation — quality matters |

To swap models, change the `model=` string in `build_llms()` (`main.py`) and `get_graph()` (`frontend/app.py`) — one line each, both marked with `TODO`.

### Product Database

`data/laptops.db` — SQLite, 28 laptops, normalized across 7 tables. Source is `data/laptops.sql`, generated by `data/build_sql.py`. Rebuild anytime with `python data/init_db.py`.

Key tables: `laptops` (main), `laptop_use_cases` (junction table for use-case tag filtering), `cpu_specs`, `gpu_specs`, `display_specs`, `ram_specs`, `storage_specs`.

Search agent filter logic: `budget * 0.8` to `budget * 1.2` price range, use-case via junction table, weight filter when `mobility == "high"`.

### Monitoring

LangSmith tracing is automatic with LangGraph — no extra code needed. Business-level metrics (recommendation success, drop-off stage) are logged explicitly via `langsmith.Client().create_feedback()` at conversation end using `state["laptop_primary"]` and `state["current_stage"]`.

## Coding Standards

**KISS — Keep It Simple.** This is the most important rule. No clever abstractions, no over-engineering. If a simpler approach works, use it. A student reading the code for the first time should be able to follow it.

**OOP — use classes for agents.** Each agent is a class. State and logic belong together, not scattered in loose functions.

**Comment everything non-obvious.** Comments explain *why* and *what*, not just *how*. Students need to understand what each block does. Every class, every method, every non-trivial block of logic gets a comment.

Example style:
```python
class IntakeAgent:
    """Collects the 4 required slots from the user before routing begins.
    Replaces the Rasa NLU layer — uses structured LLM output instead.
    """

    def __init__(self, llm):
        # Store the LLM client so every method can use it
        self.llm = llm.with_structured_output(IntakeResponse)

    def run(self, state: BuyBotState) -> dict:
        # Build the prompt with current slot state so the LLM knows what's missing
        prompt = self._build_prompt(state)
        # Call the LLM — returns both a chat message and any newly detected slot values
        response = self.llm.invoke(prompt)
        ...
```
